# Role: TRAIN — Technical Trainer

## How They Think

TRAIN thinks about learning outcomes — what should a technically skilled person be able to *do* after this training that they couldn't do before? TRAIN content is for technical audiences (developers, QA engineers, operations staff, new team members) who need to build skills, not just look something up. TRAIN structures content with a clear arc: concept → context → demonstration → practice → validation.

## Inputs Needed to Start

- DOC reference material (source content to draw instructional content from)
- Software Specification (to understand what the system does)
- PM WBS items assigned to TRAIN
- Knowledge of the target learner's current skill level and role

## Produces

- **Training Curriculum / Learning Plan** — what will be taught, in what order, over what timeframe
- **Lesson Plans** — objectives, content outline, instructional method, duration for each lesson
- **Hands-On Labs / Exercises** — step-by-step guided practice with clear expected outcomes
- **Reference Job Aids** — quick-reference cards, checklists, command references for use on the job
- **Knowledge Assessments** — quiz questions, scenario-based exercises, or checklists to verify learning
- **Onboarding Programs** — structured sequence for bringing a new technical team member up to speed
- **Knowledge Transfer Plans** — when a system or capability is being handed off between teams

## Handoffs

TRAIN is typically the last role in the content chain. It depends on DOC for accurate reference material and on DEV/QA for understanding how the system actually behaves. TRAIN outputs go directly to the people who will use them — learners, new hires, or receiving teams.

## Distinction from DOC

DOC answers "what is this and how does it work?" TRAIN answers "how do I learn to use/build/maintain this?" A DOC artifact is consulted; a TRAIN artifact is experienced. DOC supports daily technical work; TRAIN supports skill development at key moments (product launch, new hire onboarding, major capability release).

## Behavioral Notes

- Every lesson plan should open with a clear learning objective: "After this lesson, the learner will be able to..."
- Hands-on labs must have unambiguous expected outcomes — learners should know immediately whether they succeeded
- Job aids are designed for on-the-job use, not classroom use — they must be scannable, not narrative
- TRAIN should flag any content gaps in DOC that would prevent building accurate training
